The Personal Financial Tracker Handbook

Everything about installing, running, and living with PFT — written so you can follow it step by step without guessing. This single file works offline; keep it next to the app and open it any time (it lives at docs/handbook.html in the app folder).

1.What this is — and what it is NOT

A lot of people run their money life out of a spreadsheet. This tool is meant to replace that spreadsheet — or at least be the starting point you reshape, with an AI agent's help, until it fits exactly how you think about your finances.

Personal Financial Tracker (PFT) is a personal finance app that runs entirely on your own computer. It is a small web server (Python + Flask) storing everything in one SQLite database file (finance.db) in a folder you control. You use it in your browser at http://127.0.0.1:5005 — an address that only your machine can reach.

The workflow it's built around: you download CSV statements from your bank's website, import them, review and approve each transaction in a queue, and from then on the app gives you clean overviews, budgets, receipt filing, paycheck and card reconciliation, tax estimation, and Excel exports.

What it is NOT

  • Not a cloud service. No account, no signup, no sync, no telemetry, no analytics. If you unplug your internet, the app works exactly the same. Your data never leaves your machine unless you copy it somewhere yourself.
  • Not a bank-linking app. It never connects to your bank. You feed it statement files you downloaded yourself. This is a feature: there are no credentials to leak and no third-party aggregator in the middle.
  • Not financial or tax advice. It's record-keeping machinery. The Tax Estimator in particular is an estimate with clearly-listed simplifications (section 10) — a CPA or real tax software is ground truth.
  • Not multi-user. One password, one owner, one household's books.
The security model in one paragraph The app listens only on 127.0.0.1 (your own machine), requires a login password, and blocks cross-origin browser requests. The password stops someone at your unlocked laptop from browsing your finances; it is not internet-facing security, because the app is never on the internet. Your real protections are: keep your computer's own login secure, and keep backups (section 16).

Two routes: hand the whole job to an AI assistant (recommended — works even if you've never opened a terminal), or do it by hand. Either way you need Python 3.10 or newer; both routes below handle installing Python if you don't have it.

2b. Manual install — macOS, from "open Terminal"
  1. Open Terminal. Press Cmd+Space, type Terminal, press Return. (Or Applications → Utilities → Terminal.) You get a window with a text prompt — everything below is typed there.
  2. Check Python. Type python3 --version and press Return. You want Python 3.10 or newer.
    If Python is missing or too old Download the macOS installer from https://www.python.org/downloads/, run it like any app installer, then close and reopen Terminal and run python3 --version again. (If you use Homebrew: brew install python also works.) The very first time you run python3 on a fresh Mac, macOS may pop up an offer to install "command line developer tools" — accept it.
  3. Get the code. Either 
    git clone https://github.com/jberat/jby_personal_finance_tracker.git ~/personal-financial-tracker
    or download the ZIP from the repository page in your browser, double-click to unzip, and move the folder to your home folder, renamed personal-financial-tracker.
  4. Create and activate a virtual environment (keeps the app's Python packages isolated from everything else): 
    cd ~/personal-financial-tracker
python3 -m venv venv
source venv/bin/activate
    Checkpoint Your prompt now starts with (venv). If it doesn't, the source line didn't run — re-run it exactly as written.
  5. Install dependencies: 
    pip install -r requirements.txt
    This prints a lot of lines and ends without an error. If it errors, run pip install --upgrade pip and retry; if it still errors, paste the whole message into an AI assistant.
  6. Run it: 
    python3 app.py
    Leave this Terminal window open — it is the app.
    Checkpoint The terminal prints a [pft] STARTING line and then a line saying it's serving on http://127.0.0.1:5005. Open that address in your browser: you should see the login page with a single password field. If the browser says "can't connect", the app isn't running — look at the terminal for an error.

Optional, nicer startup (macOS/Linux): the repo ships two helper scripts. ./start.sh starts the app in the background (logging to app.log) and opens your browser; ./restart.sh kills any old copy — including anything squatting on port 5005 — and starts fresh. If they aren't executable yet, run chmod +x start.sh restart.sh once. Note these scripts find Python on their own and don't use the venv; if you installed inside a venv, prefer python3 app.py with the venv active, or install the requirements globally too.

2c. Manual install — Windows, from "open PowerShell"
  1. Open PowerShell. Press the Windows key, type PowerShell, press Enter. A blue-ish window with a text prompt opens.
  2. Check Python. Type python --version and press Enter. You want 3.10+.
    If Python is missing (or the Microsoft Store opens) Download the installer from https://www.python.org/downloads/. On the installer's first screen, check the box "Add Python to PATH" — this is the single most commonly missed step on Windows. Then close and reopen PowerShell and run python --version again.
  3. Get the code. Clone with git if you have it, or download the ZIP from the repository page, right-click → Extract All, and put the folder at C:\Users\<you>\personal-financial-tracker.
  4. Create and activate a virtual environment: 
    cd $env:USERPROFILE\personal-financial-tracker
python -m venv venv
venv\Scripts\activate
    If activation is blocked PowerShell may refuse to run the activation script ("running scripts is disabled"). Fix: run Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser, answer Y, then run venv\Scripts\activate again. Your prompt should now start with (venv).
  5. Install dependencies: 
    pip install -r requirements.txt
  6. Run it: 
    python app.py
    Leave the window open, then browse to http://127.0.0.1:5005.
    Checkpoint You see the login page with one password field. If not, read the PowerShell window for the error and hand it to an AI assistant.

The start.sh/restart.sh helper scripts are macOS/Linux shell scripts and don't run on Windows — on Windows you start the app with python app.py and stop it with Ctrl+C.

2d. Starting and stopping, forever after
  • Stop: press Ctrl+C in the terminal running the app (or close that terminal). Stopping never loses data — everything is written to finance.db as you go.
  • Start (macOS/Linux): open a terminal in the app folder and run ./start.sh, or activate the venv and run python3 app.py.
  • Restart after a glitch (macOS/Linux): ./restart.sh — it force-kills anything on port 5005 first, so it fixes "address already in use" too.
  • Start (Windows): open PowerShell, cd into the folder, venv\Scripts\activate, then python app.py.

Optional but recommended: the receipts pipeline's OCR and PDF handling use external tools on some systems — see section 7 for what to install if receipt scanning complains.

3.First launch — and changing the password

The first time the app runs it creates an empty finance.db, seeds a generic two-level category tree, and creates three placeholder accounts you can rename later: My Checking, My Credit Card, and Venmo. The dashboard will look bare — that's normal until your first import.

Log in

  1. Open http://127.0.0.1:5005 in your browser. You see a login page with a single password field (no username).
  2. Type changeme and submit.
    Checkpoint You land on the Dashboard, with a sidebar on the left listing Dashboard, Review Queue, Trash, Income, Expenses, Investments, Tools, and Docs & Settings. If you instead see "Wrong password.", you typed something other than changeme — it's all lowercase, no spaces.

Change the password — do this now

  1. In the left sidebar, click Docs & Settings to expand the group, then click Security.
  2. On the Change password card, fill in Current password (that's changeme), New password, and Confirm new password.
  3. Click Change password.
    Checkpoint A green "✓ Changed — use it on your next login" message appears next to the button and the fields clear. The change is effective immediately — no restart needed. If you see a red message instead, read it: "Current password is wrong" means step 2's first field, and a mismatch means the two new-password fields differ.
If you ever forget the password It's stored in a plain text file called .app_password sitting next to app.py in the app folder (the file is created the first time you change the password, and it's excluded from git). Open that file to see the password, or write a new one into it and restart the app. If the file doesn't exist and you never changed the password, the password is still changeme.
Common mistake Don't skip the password change because "it's only local." Anyone who sits at your unlocked computer can open 127.0.0.1:5005, and changeme is the first thing they'd try.

4.Try it with sample data

The app ships with three fully synthetic statements in the sample_data/ folder — a checking CSV (with paychecks, rent, utilities, transfers), a credit-card CSV (groceries, dining, subscriptions, gas, and a refund), and a Venmo CSV. Every vendor, person, and amount is invented. Use them to drive the whole pipeline once before trusting it with your real data.

4.1 — Import the checking file

  1. In the sidebar, click Tools, then Import. You land on a page with two cards: CSV Upload and Manual Entry. Click into CSV Upload.
  2. You're now on the Import CSV page. In the Account dropdown — which deliberately starts on — Select account — so you can't import against the wrong account by accident — choose My Checking (0001) — checking.
  3. Under CSV File, click the file chooser and pick sample_data/sample_checking.csv from the app folder — or just drag the CSV from your file manager and drop it anywhere on the upload card (the card highlights while you hover; the chosen filename appears under the file field either way).
  4. Click Upload & Import.
    Checkpoint You are redirected to the Review Queue with a green banner reading "Imported 26 new transaction(s). Review below." and the sidebar's Review Queue badge shows a count. If you instead get a red banner starting "Parsed 0 importable rows — this doesn't look like a checking export…", you picked the wrong account for the file (or the wrong file) — that message is the app's wrong-file guard doing its job.

4.2 — Import the credit-card and Venmo files

  1. Go back to the import page (Review Queue → ↥ Import button at the top right, or Tools → Import → CSV Upload).
  2. Import sample_data/sample_creditcard.csv against My Credit Card (0002) — credit card.
  3. Import sample_data/sample_venmo.csv against Venmo (venmo) — wallet.
    Checkpoint The credit-card import reports 40 new transactions, but only 39 land in the pending queue — the 40th row, the card's "AUTOMATIC PAYMENT - THANK YOU" line, is auto-skipped into its own strip so it can't double-count against the checking-side payment. The Venmo import reports 10.

4.3 — Work the Review Queue

Click Review Queue in the sidebar. Every imported row is here, waiting for your sign-off. Things to notice while you look around:

  • Most rows arrived with a suggested vendor (cleaned up from the raw bank text — e.g. TST* HARBORLIGHT BISTRO became "Harborlight Bistro") and many with a suggested category.
  • Some rows carry a small pill explaining a flag: Internal transfer on the savings transfers and the Venmo cash-out, CC payment on the checking-side autopay rows (kept as transfers so the Reconcile Card tool can use them).
  • Click any field (vendor, category, amount, date) to edit it in place; a small popup with Save / Cancel appears for the bigger edits.
  1. Approve a few rows one at a time with the green ✓ button at the right of each row (hover text: "Approve — move to transactions"). Discard is the ✕ next to it — discarded rows are kept in a separate "discarded" view, not destroyed.
  2. Approve the rest. (Keyboard: with a row focused, Cmd+Enter approves.)
    Checkpoint The queue empties and the sidebar badge disappears. Approved rows are now real ledger transactions.

4.4 — Look at what you built

  1. Dashboard — year-to-date cards and recent activity now have numbers.
  2. Expenses → Overview — a category-by-month pivot. Click any L1 row (e.g. Food & Dining) to expand its L2 breakdown.
  3. Income → Overview — the paychecks and Zelle income.
  4. Expenses → By Vendor — notice the fake subscriptions (Streamflix, Tunebird) lining up month after month; this is the subscription-creep view.

4.5 — Clean up when you're done

  1. Go to Tools → Import → CSV Upload. The Recent Imports table on the right lists your three batches.
  2. Click Undo on each sample batch and confirm. This permanently removes the batch's staged rows and any transactions approved from them, and frees the duplicate-detection keys — a complete rollback to the empty state.
    Checkpoint The Dashboard is empty again. You're ready for real data.

5.The core loop — your monthly routine

Everything else in the app is optional. This loop is the product: about 20–30 minutes, once a month, when your statements close.

Step 1 — Download CSVs from your bank

  1. Log in to your bank's website (yourself, in your own browser — the app never does this).
  2. Find the transaction/statement download and choose CSV format (sometimes labeled "spreadsheet"). For Chase: open the account, find the download icon above the activity list, pick a date range and CSV. For Venmo: Statements → Download CSV.
  3. Pick a range that overlaps your last download rather than trying to butt ranges perfectly — overlap is safe because duplicate detection flags anything already imported.
Common mistake Downloading PDF statements. The importer reads CSVs only. If your bank offers "PDF" and "CSV/Excel", you want CSV. (Excel .xlsx also won't import — if that's all you have, open it and save as CSV, keeping plain numbers in the amount column.)

Step 2 — Import each file

  1. Tools → Import → CSV Upload.
  2. Select the matching account in the Account dropdown — checking file to a checking account, card file to that card, Venmo export to the Venmo account. The dropdown never pre-selects an account, precisely so you look at this every time.
  3. Choose the file — click to browse, or drag the CSV onto the upload card (dropping only fills the file field; nothing imports until you click the button) — then click Upload & Import.
Checkpoint A green banner: "Imported N new transaction(s)" — possibly with "· X duplicate(s) flagged for review" if your ranges overlapped (expected and fine). A red "Parsed 0 importable rows…" banner names the exact problem: the header-sniff message ("this doesn't look like a checking export (missing the "Posting Date" column)") means file/account mismatch; an "unparseable Amount" count with an example like "$1,063.03" means the file was hand-edited into currency formatting.

Two side effects worth knowing: the raw CSV is archived automatically into imports/checking/, imports/creditcards/, or imports/other/ with a timestamp prefix (permanent import history), and every batch gets an Undo button in Recent Imports.

Step 3 — Work the Review Queue

Approve (✓), edit-then-approve, or discard (✕) each row. The queue is where data quality happens:

  • Fix vendors and categories here, not in the ledger later. If you notice yourself fixing the same vendor every month, teach the auto-categorizer instead (section 6).
  • Flagged rows are segregated, not hidden. Duplicates and auto-skipped rows (card-side "Payment Thank You" lines, Venmo rows that would double-count) sit in their own strips with explanatory pills. Note the asymmetry: the CHECKING-side payment row is NOT skipped — it imports as a pending "CC Payment" transfer, because the Reconcile Card tool needs it to tie statements. Confirm the flags are right; restore anything wrongly flagged.
  • Signs are handled for you. The queue shows amounts with the bank's raw sign; on approve, the app converts to its own convention (for expense rows: positive = money out, negative = a refund/credit). A cash-in over $500 on checking or Venmo defaults to income; under $500 it defaults to a contra-expense (a refund) — re-type it in the queue if the guess is wrong.

Step 4 — Sweep receipts, then reconcile

  1. Drop the month's receipts into the receipts inbox and run a scan (section 7).
  2. If you use the Payroll tool, reconcile the month's paychecks (section 9).
  3. After the card autopay has imported and been approved, run Reconcile Card (section 11).
  4. Once a quarter: skim Tools → Data Cleanup (section 15) and glance at Actuals-vs-Budget (section 8).

5a.Venmo & P2P payments — why they get their own importer

Why your bank statement isn't the full story

Your bank only ever sees Venmo as lump transfers: a "VENMO CASHOUT" deposit when you move money out of Venmo, or a "VENMO PAYMENT" pull when Venmo tops up from your bank. That is not the story of what happened. One cashout can bundle ten individual payments — the rent split, the concert tickets, someone paying you back for dinner — and payments funded by your Venmo balance never touch the bank at all, so the bank statement doesn't even hint they exist.

That's why PFT imports the Venmo statement itself: every individual payment and charge comes in as its own row, with the counterparty as the vendor (the person's name, not "VENMO") and the payment note carried along — so your books say "paid Riley $71.30, utilities my half," not "moved a lump to Venmo." Getting the file: log in at venmo.com and download the CSV for a date range — the export lives under Statements (the site moves things around, but it's the statement/CSV download you want). Import it against the Venmo account (section 5, same flow as any other CSV).

The double-counting rule

Warning — the one rule that keeps P2P books honest If you import both your bank statement AND your Venmo statement, the same dollars appear in two places. The bank-side "VENMO" rows (cashouts and top-ups) are transfers between your own accounts — they must be treated as transfers or discarded in review, never categorized as income or expense. The individual Venmo rows carry the real income/expense categorization. Break this rule and every Venmo dollar counts twice.

Here's exactly what that looks like in the review queue, and what to click:

  1. Import your checking CSV. Any row whose description contains "VENMO" is automatically segregated out of the pending queue into the skipped strip, wearing a Venmo pill — the importer knows these are lump transfers and never auto-imports them.
  2. Leave them there (they'll never enter your ledger), or discard them with the ✕ button to tidy the strip. If one of them genuinely isn't a transfer, restore it with ✓ and re-type it before approving.
  3. Import the Venmo CSV. Individual payments arrive as pending expense/income rows named after the counterparty; the Venmo-to-bank cashouts (Standard/Instant Transfer, Deposit rows) arrive typed as transfers with an Internal transfer pill — approve those as-is, they're neutral on your income and spending. Rows that weren't complete at export time (Pending, Issued, Failed) are auto-segregated with a Venmo status: … pill; don't approve those — re-import a later statement once they complete.
Checkpoint After both imports, search your Expenses and Income transactions for "Venmo": you should see people's names (the individual payments), and no lump "VENMO CASHOUT"-style rows as income or expense. The only bank-side Venmo traces in your ledger should be transfers — or nothing at all. If a lump Venmo row shows up as income or expense, delete it (it goes to Trash) — the individual rows already tell that story.

The same pattern applies to any P2P or wallet app. Cash App and PayPal work like Venmo: a balance that settles to your bank in lumps, so the app's own statement holds the real transactions and the bank-side rows are transfers (PayPal rows are deliberately not auto-skipped by the checking importer, since PayPal often rides along on real merchant purchases — review those by hand). Zelle is the exception worth knowing: it's bank-to-bank with no balance and no separate statement, so the Zelle row on your bank statement usually is the real transaction — the importer keeps it and even pulls the counterparty's name out of the description for you.

5b.Adding accounts

A fresh install ships with three placeholder accounts (My Checking, My Credit Card, Venmo). Real life has more — a second card, a joint checking account, a cash envelope. You add and manage all of them inside the app; no code editing involved.

Adding an account, click by click

  1. In the left sidebar, click Docs & Settings to expand the group, then click Accounts.
  2. In the Add an Account card, type a Name you'll recognize in dropdowns — e.g. My Sapphire Card.
  3. Pick a Type. This decides how transactions get in:
    • Checking — CSV imports in the Chase checking format.
    • Credit card — CSV imports in the Chase credit-card format.
    • Digital wallet — CSV imports in the Venmo statement format.
    • Other — no importer; transactions are entered by hand via Tools → Import → Manual Entry. Use it for cash, HSA cards, anything without a usable CSV.
    If your bank's CSV isn't one of those formats, still pick the right type here, then have an AI assistant build the importer — section 14.
  4. Optionally type the card/account's last 4 in the number field. It's display-only — it appears next to the name in dropdowns so two similar accounts are easy to tell apart.
  5. Credit cards only: two extra fields appear.
    • Statement close day (required) — the day of month your statement cycle closes, same day every month (it's printed on every statement, e.g. "Statement closing date: 06/25" → enter 25). This drives statement-date assignment on CSV import and powers the Reconcile Card wizard. For days 29–31, months without that day close on the month's last day.
    • Payment due day (optional) — the day of month the payment is due, used to estimate payment dates on imported charges.
    Existing credit-card accounts can add or correct these any time: edit the two Billing days boxes in the account's table row and click Save.
  6. Click Add Account.
    Checkpoint A green banner confirms the add, the account appears in the Your Accounts table, and (for the three importable types) it's now in the Account dropdown on the Import CSV page.

Rename, deactivate — but never delete

  • Rename — edit the name right in the table row and click Rename. History follows automatically (transactions point at the account, not its name). Duplicate names are rejected, capitalization included.
  • Deactivate — hides the account from the import dropdown and every new-transaction picker. Everything it already holds keeps rendering on every page. Closed a card? Deactivate it — its years of history stay intact. Reactivate brings it back any time.
  • There is no Delete, on purpose. Transactions reference their account forever; deleting one would orphan its history. Deactivate is the answer.

6.Categories, auto-categorization & vendor cleanup

Every transaction gets two category levels: L1 (the broad group, like "Food & Dining") and L2 (the specific bucket, like "Groceries"). The app ships with a sensible starter tree for expenses, income, and transfers.

Viewing and adding categories in the app

  1. Sidebar → Docs & Settings → Categorization. The full L1/L2 trees are listed by type (expense / income / transfer).
  2. To add: use the Add categories card at the top. Either create a brand-new L1 with its first L2s (Add L1 + L2s — a new L1 requires at least one L2), or add L2s under an existing L1 (Add L2s).
    Checkpoint A green confirmation ("Added to existing L1 — 2 L2(s) added — reloading…") and the tree refreshes. Additions are usable immediately in every dropdown and survive restarts.
Renames and removals are different The in-app page only adds. Renaming or removing categories is a two-part surgery, because transactions store category names as plain text — renaming the tree without updating existing rows orphans their history. Use the ready-made AI prompt ("Recipe 2") in docs/customize_with_ai.md, which edits the starter tree in categories.py and cascades the rename across existing transactions. Back up first.

How a row gets its suggested vendor and category

Understanding the pipeline makes it obvious what to change when a suggestion is wrong. At import, each row passes through, in order:

  1. Vendor cleanup (vendor_rules.py plus a per-importer vendor map): shared rules strip processor prefixes (SQ *, TST*), store numbers, and reference-number noise; known fragments map to canonical names; Zelle rows get the counterparty's name pulled out of the description. Matching is word-boundary aware, so "APPLE" can't hijack "APPLEBEES".
  2. Type classification (per importer): expense / income / transfer, using the bank's own type column, transfer patterns, and the $500 cash-in threshold.
  3. Skip flags: card-side bill-payment rows ("AUTOMATIC PAYMENT", "PAYMENT THANK YOU", autopay variants) and Venmo rows on the checking side are auto-segregated so nothing double-counts. The checking-side card payment itself is not skipped — it imports as a pending "CC Payment" transfer (see section 5).
  4. Bank category hints (credit card only): Chase's own category column maps to a starter L1/L2 pair.
  5. Review rules (the ladder in routes_review.py, _apply_review_rules): first-match-wins vendor/description patterns that set L1/L2 — e.g. airlines to Travel/Airfare, "Costco but not Costco Gas" to Groceries. This is the layer you'll extend most.

Teaching it your own vendors — the AI recipe

Collect the fixes you keep making by hand, then paste this into an AI assistant along with the files it names (this is Recipe 3 from docs/customize_with_ai.md):

I'm using Personal Financial Tracker, a local Flask + SQLite personal
finance app. I want to add auto-categorization rules so imports come in
pre-categorized. Two files are involved:
- vendor_rules.py — cleans raw bank descriptions into canonical vendor
  names (ordered substring/fragment matching).
- The auto-categorization rules applied at import time — an ordered,
  first-match-wins ladder that maps vendor/description (and sometimes
  amount) to (trx_type, l1_category, l2_category). Find where the
  existing rules live (the import pipeline applies them right after
  parsing) and follow their exact style.

Here are my rules, written as "raw description or vendor → what I want":
<EXAMPLES, e.g.:
- Every "FOO LLC" charge → Food & Dining / Takeout
- Anything containing "SQ *BLUE BOTTLE" → vendor "Blue Bottle Coffee",
  category Food & Dining / Coffee Shops
- "CITY OF SPRINGFIELD UTIL" → Housing / Utilities>

How to do it:
1. Where a raw description needs a nicer vendor name, add a cleanup rule
   in vendor_rules.py. Use word-boundary fragment matching the way the
   existing rules do, so short fragments don't accidentally match inside
   longer words.
2. Add category rules to the import-time rules ladder, placed so
   more-specific rules come before general ones (first match wins).
3. My category names must EXACTLY match names in my category tree
   (categories.py / the categories table) — verify each one and tell me
   if any of my requested categories don't exist.

These rules apply to FUTURE imports only, which is fine. I have backed up
the app folder first. Only edit the rules — don't touch the database or
the importers. When done, tell me how to test.

Do this once a quarter and the review queue gets progressively boring — which is the goal. For one-off vendor messes already in your ledger, use Data Cleanup's global vendor rename instead (section 15).

7.Receipts, end to end

The receipts pipeline turns a shoebox of PDFs and phone photos into receipts that are matched to transactions and filed in an organized folder tree. The payoff: any transaction can show its receipt in one click, forever — warranty claims, tax questions, HSA reimbursements years later.

One-time setup check

OCR and PDF handling use external tools on some systems. If the scan step later reports extraction failures: on macOS, install brew install poppler tesseract (poppler reads PDFs; tesseract is the OCR fallback — macOS also has a built-in OCR path that usually works out of the box). On Windows/Linux, install your platform's poppler-utils and tesseract packages. iPhone HEIC photos are converted automatically.

The flow

  1. Drop files in the inbox. The inbox is an ordinary folder: receipts/inbox/ inside the app folder. PDFs, JPG/PNG/HEIC photos — don't bother renaming anything. Emailed receipts: print-to-PDF or download the attachment. Paper: photograph flat, decent light, whole receipt in frame including the total.
  2. Scan. Sidebar → Tools → Receipts, then click ⟳ Scan inbox at the top. The app reads each file, extracts vendor / amount / date, and scores candidate transactions from your ledger on those three signals.
    Checkpoint The status line under the button reports what the scan did, and the page's counters update: high-confidence matches file themselves; medium-confidence ones land in the review pile; no-match receipts become orphans.
  3. Review the queue. Click Start review →. One receipt at a time: the file preview on one side, the proposed transaction on the other. Actions (all keyboard-driven; defaults shown, editable in Docs & Settings → Shortcuts): Confirm & file (Enter), skip (s), not-a-receipt (n), not-this-match/search (r), link to another transaction (l), open split panel (x), discard duplicate (d).
  4. Place orphans when their transactions arrive. Orphans aren't stuck — every scan re-checks them against the current ledger, so a receipt that arrived before its bank transaction matches up on a later scan. You can also attach one manually from the transaction's detail page.

Where files end up

receipts/filed/<year>/<L1 category>/<L2 category>/<month>/"YYMMDD Vendor.pdf"

Year/month come from the transaction's date and the folders from its categories, so the filing mirrors your ledger. Collisions get a numeric suffix; nothing is ever overwritten, and nothing is ever hard-deleted by the app (rejected files go to a receipts trash view where you can restore or truly delete).

Honest expectations Receipts are this tool's weakest flow — OCR is genuinely hit-or-miss on photos. Expect orphans and the occasional bad match, especially when the matching transaction hasn't been imported yet. The review screen exists to catch exactly that; budget a little patience.
Common mistakes Scanning before importing. Matching runs against your ledger — import and approve the bank statement first, then scan receipts. Bad photos. Glare, crumpled paper, or a cropped-off total are the main causes of wrong amounts and orphans; the confidence tiers usually catch it, but better photos mean less review. Foreign-currency receipts can't be tied to the USD charge automatically — expect to place travel receipts by hand.

Mark receipt-less-by-nature transactions (bank fees, the $4 coffee) with the no-receipt-needed flag — from the transaction detail page or in bulk from Data Cleanup — so the "missing receipts" list stays an honest to-do list. Deeper detail on extraction and matching internals: docs/receipts_guide.md.

8.Budgets & the Actuals-vs-Budget report

Setting budget values

  1. Sidebar → Docs & Settings → Assumptions → Budget Values.
  2. Pick the year at the top.
  3. For each category you actually want to control, enter an annual amount — either one flat number on the L1 row, or expand an L1 and budget its L2s individually.
    The rollup rule As soon as any L2 under an L1 has a non-zero budget, that L1's effective budget becomes the sum of its L2 rows — the flat L1 input locks and shows the rolled-up total. So budget an L1 flat, or by its L2s, but they don't add together.

Income categories can be budgeted on the same page — they get their own section in the report, with the goodness inverted (ahead of pace = green). Advice on what to budget (few lines, only where behavior moves the number) is in docs/finance_tips.md.

Reading the report

  1. Sidebar → Tools → All Tools → Actuals vs. Budget (URL: /tools/actuals-vs-budget).
  2. Choose your view:
    • Pace (default) — year-to-date actuals vs a prorated budget (annual ÷ 12 × complete months). Answers "am I on pace?" Rows sort worst-first.
    • Annual — YTD actuals vs the full annual budget, with the year-elapsed percentage as a reference marker. Answers "how much room is left?"
  3. Read the colors: green = at/under pace, yellow = up to 25% over pace, red = more than 25% over. L1 rows with L2-level budgets expand to show the L2 detail. Actuals are pinned to whole-month boundaries so the proration is a fair comparison.

9.Payroll reconciliation — booking your paychecks at gross

Why this exists (the gross-vs-net problem, for non-accountants)

Say you earn $4,000 in a pay period. Your employer withholds, for example, $600 federal tax, $180 state tax, $306 Social Security + Medicare, and puts $400 into your 401(k) — and deposits the remaining $2,514 in your bank. Your bank statement — and therefore your import — only ever sees the $2,514. Left alone, your books would say you earned $2,514, your income would look ~37% smaller than it is, and your single biggest expense category (taxes) would be completely invisible.

The Payroll tool fixes that by splitting each imported deposit into its true parts: +$4,000 gross wages, −$600 federal tax, −$180 state tax, −$306 FICA, −$400 pre-tax retirement. Those five lines still sum to $2,514 — your cash totals don't change by a cent — but the Income pages now show real earnings and real taxes paid. Gross books under Salary & Wages; taxes and pre-tax health book as negative "contra-income" lines under Taxes and Pre-Tax Deductions. The retirement amount is different: it's your own money moving into your own retirement account — savings, not income lost — so it books as a transfer (and can flow straight into the Investments module, below). It's employee-side only — no employer-cost bookkeeping.

Post-tax lines — Roth and garnishments

Paychecks can also carry deductions taken after tax, and the tool books each by what it really is:

  • Roth / post-tax retirement — a savings transfer, exactly like pre-tax retirement: your own (already-taxed) money moving into your own retirement account. It books as a transfer sharing the same Retirement account (Investments) target as the pre-tax amount — if a paycheck has both, you get two transfer rows into the same account, and each can sync into Investments as its own contribution.
  • Garnishment / other post-tax deduction — money you earned and then had taken (wage garnishment, union dues, and the like). It books as contra-income (a negative income row) under its own category, Post-Tax Deductions / Garnishments & Other — not under Taxes, and not hidden in the generic plug — so your gross earnings stay truthful and the deduction is visible by name.

The tie check everywhere is now gross − taxes − pre-tax − post-tax − other = net. The Gusto CSV parser also recognizes Roth and garnishment columns (e.g. "Roth 401(k) (Employee Deduction)", "Garnishment (Employee Deduction)") and buckets them automatically. Undo reverses post-tax rows the same as everything else, including any Roth contribution synced into Investments (same still-untouched rule as pre-tax).

Tracking your 401(k) in Investments — set the account up FIRST If you contribute to a 401(k)/retirement plan and want it tracked in Investments, create the investment account before reconciling: Investments → Overview → add an account (name it, group Retirement — see the Investments section for the full setup steps). Then run payroll reconciliation and, on the reconciliation page, check the Retirement account (Investments) dropdown — with a single Retirement-group account, "auto" picks it for you. On execute you get, for a $400 contribution: a $400 transfer transaction (category Retirement / your-account-name, dated the pay date) in your transactions, and a $400 contribution event in that Investments account on the same date — created by the same Sync engine every investment transfer uses, so it can never double-book.

Reconciled before creating the account? Nothing is lost: the transfer row still books, under the generic Retirement / Retirement Plan category, and nothing lands in Investments. To fix it afterwards, create the investment account, then either (a) undo the reconciliation, pick the account in the dropdown, and re-execute, or (b) open the transfer transaction, change its L2 category to the account's exact name, and press Sync on Investments → Transactions — the sync pulls any qualifying transfer it hasn't seen yet.

Path 1 — Gusto CSV upload

  1. In Gusto: Reports → Payroll Journal Report → CSV; download it.
  2. In the app: sidebar → Tools → Payroll. On the Import a Gusto CSV card, choose the file and click Parse + review. (The file auto-saves to imports/payroll/ for permanent history — safe to delete your download afterwards.)
  3. Each pay period in the file becomes a draft reconciliation. Open one: you see the parsed employee-side numbers (gross, each tax, pre-tax deductions, net), the matched bank deposit if one is already imported, and the exact rows that will be booked.
  4. Check the tie line — the planned rows must sum to net pay ("✓ ties"). Then click Confirm + book gross-up rows.
    Checkpoint The status pill flips to "✓ Reconciled." If a matching net deposit existed, it's now a split parent whose children are the gross/tax/deduction lines; check Income → Transactions and you'll see gross wages and negative tax lines for that pay date. If the button is disabled with "The math must tie before executing", a number is off — fix it or use the plug field.

Path 2 — Manual paystub true-up (any employer)

  1. Tools → Payroll → Open true-up form → (the "Enter a paystub manually" card). You're on the Paystub true-up page.
  2. Pick the Entry basis: One paycheck (if the matching net deposit is already imported, the tool splits it in place) or Year-to-date totals (annual true-up) (one entry books the whole year's gross-up as standalone rows — handy if you'd rather do this once a year from your final paystub or W-2).
  3. Type the numbers straight off the stub: gross pay, federal / state withholding, FICA (one combined line — if your stub splits Social Security and Medicare, add them), other payroll taxes (PFML / SDI / local), pre-tax retirement, health/other pre-tax, Roth / post-tax retirement, garnishment / other post-tax, and net pay.
  4. Watch the live tie check next to Net pay: it continuously verifies gross − taxes − pre-tax − post-tax − other = net. If your stub has some line the form doesn't (rounding, a one-off), put it in Other adjustment (plug) so the math ties.
  5. Click Save draft →, review the plan on the next page exactly as in Path 1, and Confirm + book gross-up rows.
Undo is always available Open any reconciled entry and click Undo reconciliation: every booked row is deleted and, if a deposit was split, it's restored to its original single-row state. A contribution the reconciliation synced into Investments is removed too, as long as it's still untouched there (if you've since moved or revalued that money, the event stays and you manage it from the Investments ledger). The tool records exactly what it created, so undo is complete.
Common mistakes Reconciling before importing the bank statement — then there's no deposit to split, and the rows book standalone; if the deposit imports later you'll have both. Import and approve the deposit first. Entering employer-side numbers (employer 401k match, employer taxes) — this tool is employee-side only; employer costs aren't your income or your expense.

10.Tax Estimator

Sidebar → Tools → Tax Estimator. A six-step walkthrough that turns a handful of numbers into an estimate of your 2026 federal and state income tax — and, once you enter what you've already paid, into "what you still owe (or your refund)" with a simple quarterly split. It is completely stateless: nothing you type is saved, sent, or written to the database; close the tab and it's gone.

The walkthrough

  1. Read me first. The caveats card (worth actually reading — the full list is below). Click I understand — start.
  2. About you. Pick a filing status (Single / Married filing jointly / Married filing separately / Head of household) and your state. The state dropdown is grouped: No income tax, Flat rate — supported, and Not yet supported (federal only) — picking an unsupported state is fine (each carries a "⚠ federal only" marker and an inline note appears the moment you pick one), you still get the full federal estimate.
  3. Your income types. Toggle what applies — W-2 job, self-employment/1099, interest & dividends, capital gains, other. You'll only be asked for numbers on what you toggle. Toggling self-employment also asks how it's structured: sole proprietor / single-member LLC (the default — SE tax applies to your Schedule C line 31 net profit, and the half-SE-tax deduction is taken automatically) or S-corp owner paying yourself W-2 (salary goes in the W-2 fields; owner distributions get their own field and are not subject to SE tax). The results notes state which treatment was applied.
  4. Your deductions. Standard deduction (the default; the card shows the 2026 amount for your filing status) or itemize (you'll enter one total). Also toggle traditional IRA / HSA contributions made outside payroll.
    The double-count trap 401(k) deferrals and payroll health premiums are usually already excluded from your W-2 Box 1 wages — they appear in Box 12 (code D for 401(k)), not Box 1. Self-check on last year's W-2: if Box 3 (Social Security wages) is bigger than Box 1, the difference is your pre-tax deferrals, already excluded. Do not enter them again as deductions here — that double-counts. The IRA/HSA fields are for contributions made outside payroll only. (The tool repeats this warning in its results notes.)
  5. Enter your numbers. Only the fields your toggles selected, each with a hint of where it comes from (W-2 Box 1, 1099-DIV box 1a/1b, etc.). Click See my estimate →.
  6. Your estimate. You get: the full income-to-taxable walk (total income → AGI → taxable), federal tax broken into ordinary tax, capital-gains tax by 0/15/20% band, self-employment tax, additional Medicare, and NIIT, the state line, marginal and effective rates, and a notes list of every approximation that applied to your inputs. In Payments & what's left, enter federal/state withholding (W-2 Boxes 2/17) and any estimated payments, click Recalculate, and read the remaining-owed / refund line with the suggested quarterly split.

What the federal math includes

  • 2026 ordinary-income brackets for all four filing statuses, and the 2026 standard deduction.
  • Long-term capital gains and qualified dividends taxed by the stacking method (they sit on top of ordinary income and get 0/15/20% by where they land).
  • Self-employment tax (Schedule SE) with the wage-base interaction against your W-2 wages, and the half-SE-tax deduction fed back into AGI.
  • Additional Medicare tax (0.9%) and a simplified NIIT (3.8%).
  • Credits subtracted dollar-for-dollar from federal income tax (you enter one total).

State support

  • No-income-tax states — AK, FL, NV, SD, TN, TX, WA, WY: state tax $0. New Hampshire also shows $0 (its interest & dividends tax was repealed). Washington's excise tax on large capital gains is not modeled.
  • Flat-tax states (approximated) — AZ, CO, GA, ID, IL, IN, KY, MI, MS, NC, PA, UT: computed as the state's flat rate × a federal base (federal taxable income or federal AGI, whichever is the closer proxy for that state). Each shows a specific approximation note — several of these states have scheduled rate cuts, and state-specific deductions/exemptions are not modeled, so some estimates overstate.
  • Everything else (graduated brackets) — not yet supported; the federal estimate still computes and the tool says so plainly. More states are coming.

The complete honest caveats list

Not tax advice — and these are the known simplifications
  • This is an estimate, not tax advice; no CPA was involved. Verify anything that matters with a professional or real tax software.
  • Rates are for tax year 2026 and should be verified before relying on results — several state rates have scheduled changes.
  • No AMT (alternative minimum tax).
  • No QBI deduction (the 20% pass-through deduction for self-employment income).
  • No credit phase-outs — your credits total is subtracted dollar-for-dollar, which overstates credits that phase out at your income.
  • Capital losses are floored at $0 — no loss netting and no $3,000 ordinary-income offset. If you have net losses, your real tax is lower than shown.
  • Self-employment losses are excluded entirely — a negative SE amount produces no SE tax and does not reduce your other income, so with a real SE loss your actual tax is lower than shown. The results notes say so when this applies.
  • State math is flat-rate × a federal base — no state-specific deductions, exemptions, credits, or local/county/city taxes (which are material in e.g. Indiana counties, Michigan cities, Pennsylvania localities).
  • The quarterly split is simply remainder ÷ 4 — it is not a safe-harbor calculation and doesn't account for payments already being late in the year.
  • NIIT uses MAGI ≈ AGI and gross investment income (no investment expenses netted).

11.Reconcile Card — closing the credit-card cycle

Every month your checking account pays the card, and that payment settles a specific set of charges. Reconcile Card ties each payment to exactly the charges it settled, proves the math to the penny, and diagnoses anything in transit. Run it once a month, after the autopay has imported and been approved.

What it needs to work

Setup prerequisite: set the card's statement close day. The wizard buckets charges into statement cycles using the close day stored on the credit-card account — go to Docs & Settings → Accounts (section 5b) and fill in the Billing days for each card. Without it, the wizard falls back to post dates and can't derive statement cycles.

A card payment, to this tool, is a transfer transaction with L1 category Credit Card Payment whose L2 names the card. The checking importer creates these automatically from autopay/payment rows (and defaults the L2 when you have exactly one credit-card account — otherwise you pick the card in the review queue). In every type dropdown these rows display as CC Payment — a display distinction over the same stored transfer type, so card payments never blur together with investment transfers. The card-side "PAYMENT THANK YOU" rows stay skipped so nothing double-counts.

The four dates on a transaction

Card transactions carry four dates, and each answers a different question:

  • Transaction date — when you swiped.
  • Post date — when the bank recorded it.
  • Statement date — the statement close the charge belongs to. The importer assigns it from the account's close day (post-date basis); the wizard trues it up on confirm for boundary charges that landed a cycle over. All four are editable on the transaction detail page.
  • Payment date — when the cash actually left checking to settle the charge. Set by the wizard on confirm; drives cash-basis month rollups.

The wizard buckets by statement date when present, falling back to post date only for rows that don't have one.

The monthly ritual

  1. Sidebar → Tools → Reconcile Card. The Open payments — not yet reconciled table lists card payments awaiting reconciliation; below it, Open charges shows what's accruing toward the next payment. If a payment shows Set card → instead of a Preview button, open it and set its L2 to the card's name first.
  2. Click Preview → on a payment. The detail page proposes the set of charges (by statement cycle) that this payment settled, sums them, and shows the residual — proposed charges vs the payment amount.
  3. If the residual is $0.00 (green), click CONFIRM — settle N charges.
    Checkpoint The charges lock as settled, and their payment and statement dates true-up to the actual cycle. Nothing writes until a $0.00 residual is confirmed — the confirm button doesn't exist otherwise.
  4. If the residual is not zero, read the diagnosis under it. The tool looks for the usual suspects and offers one-click fixes: charges from the edge of the cycle that belong in this settlement ("include them, then work the remainder"), a charge posted after close (in transit), a still-unapproved row sitting in the review queue that matches the residual, or a prior-cycle carry. Work the suggestions until the residual reads $0.00, then confirm.

Mistakes are cheap: Unwind this settlement on a reconciled payment reverts every charge to its prior payment/statement dates.

If you carry a balance

Nothing here assumes you pay in full. Interest is just another charge: it imports as a normal expense, lands on its statement by the close day, and the tie includes it like any other line. If a payment is smaller than the cycle's charges, the wizard shows the unpaid remainder as a carried balance — a diagnosis, not an error. To reconcile a partial payment, exclude the charges it didn't cover (they stay open for the next payment's preview) until the residual reads $0.00, then confirm.

Common mistake Running this before the month's card statement rows are imported and approved. The tool matches against your ledger — if the residual won't close, the missing charge is usually still in the review queue or in a statement you haven't imported yet.

12.Investments (experimental)

The sidebar's Investments group is an optional module that tracks investment accounts with a lot-based engine: every dollar of principal you deploy becomes a lot with an origin date; contributions, withdrawals, moves between accounts, and value snapshots are events on those lots; and from that history the app computes both XIRR (money-weighted return — your personal return including the timing of your contributions) and TWR (time-weighted return — comparable to a fund's published number).

Set expectations first

Experimental means experimental This module works, but it's the newest and least battle-tested part of the app, and it operates largely independently of the rest: investment accounts and events live in their own tables, and the module doesn't reconcile itself against your bank imports beyond optional transfer syncing. Treat it as a bonus notebook for tracking your portfolio — not as the same class of machinery as the import/review ledger. Expect rough edges.

Getting started

  1. Read the module's own guide first: Investments → How it works in the sidebar — it explains lots, events, and the return math in detail.
  2. Add your investment accounts from the Investments overview page.
  3. Record contributions/withdrawals/moves on the Transfer page (or bulk-load history via Investments → Import, which provides a CSV template).
  4. Enter a value snapshot per account whenever you have a statement — quarterly is plenty. Snapshots are what turn cash-flow history into return numbers.

14.Adding YOUR bank — the AI importer recipe

PFT ships with three importers: Chase checking, Chase credit card, and Venmo. If you bank elsewhere, you have an AI assistant build an importer from a sample of your bank's CSV. This is the most common customization, it takes a few minutes, and it works remarkably well because bank CSVs are simple and the app's importer pattern is deliberately uniform (every importer is one file that parses a CSV into the same dict shape).

Step 1 — Sanitize a sample CSV (do not skip)

You're about to hand a chunk of a bank statement to an AI service, so strip anything identifying first:

  1. Download a CSV from your bank and open it in a spreadsheet or text editor.
  2. Keep the header row exactly as-is — the AI needs the real column names.
  3. Keep 10–20 real data rows so the AI sees genuine date formats, amount signs, and description quirks.
  4. Replace anything identifying: account digits (...1234 → ...0000), your name, other people's names in transfer descriptions, addresses. Change the digits, keep the format identical (same length, same punctuation).
  5. Save as sample_sanitized.csv.

Step 2 — Know your network's quirks (tell the AI which apply)

The single most common importer bug is a flipped sign convention, and it varies by issuer — check your sample against this table and mention what you see in the prompt:

Export typeWhat to expect
American ExpressAmounts are usually positive for charges (the opposite of Chase); payments/credits negative. Often a single date column, sometimes an extended details column with the merchant address glued on.
Mastercard / Visa issuers (Citi, Capital One, US Bank…) Conventions vary per issuer, not per network. Some use one signed Amount column (charges negative), others use separate Debit and Credit columns that must be merged into one signed amount. Check which yours is.
Checking / debit exportsUsually one signed amount column (deposits positive), often only a single date. Some banks add a running Balance column (ignore it) or a type column worth mapping to expense/income/transfer. Watch for transfer/autopay descriptions that should be auto-flagged so they don't double-count against your card imports.
Hand-edited / Excel-exported filesCurrency-formatted amounts like $1,063.03 or (45.00) for negatives. The app's amount parser tolerates most of this, but keep amounts as plain numbers where you can.

Attach or paste three things into the AI conversation: your sample_sanitized.csv, the file importers/chase_cc.py from the app folder (the reference pattern), and this block:

finance app. I want to add an importer for a new bank: <BANK NAME,
e.g. "Ally Bank checking">.

Attached:
1. sample_sanitized.csv — a sanitized sample of this bank's CSV export
   (real column headers, real formats; identifying digits scrambled).

   pattern exactly.

<e.g. "charges appear POSITIVE like Amex" / "separate Debit and Credit

  db) function returning (rows, dupe_count, skipped), where each row dict

Constraints: I have backed up the folder, but still — only create the new
importer file and the minimal registration change. Do not modify the

Step 4 — Test it

  1. Restart the app (./restart.sh, or Ctrl+C and start again).
  2. Import a real (un-sanitized) statement from the new bank against its account.
  3. In the review queue, check the three things that go wrong: dates look right, amount directions are right (charges as expenses, deposits as income — "all my deposits show as expenses" means the sign convention is flipped; a one-line fix, tell the AI), and vendor names are readable.

15.Data Cleanup & Trash

Data Cleanup

Sidebar → Tools → All Tools → Data Cleanup (URL: /tools/cleanup). The page re-scans your ledger on every load and presents findings in six groups, each reviewed and fixed one item at a time:

  • Vendors — names that still contain store numbers, reference IDs, or other noise, each with a suggested clean name; saving renames the vendor globally across your ledger.
  • Tags — suspected duplicates — case/space/prefix variants of the same tag, with a one-click merge into the more-used one.
  • Missing dates — rows missing payment/statement/post dates, with a one-click fill-from-transaction-date.
  • Possible duplicates — same account, day, amount, and description across different import batches (repeat charges within one statement, like two subway swipes, are deliberately not flagged).
  • Categories to re-pick — rows whose category is missing or no longer in your tree, with inline dropdowns.
  • Missing receipts — active expenses with no receipt and no no-receipt-needed flag; resolve by inboxing the receipt or clicking "No receipt needed" right there.

Every finding has an Ignore button that hides it forever (a counter at the top lets you clear the ignore list and un-hide everything). Skimming this page quarterly keeps entropy from accumulating.

Trash

Deleting a transaction anywhere in the app is a soft delete: the row moves to Trash (sidebar, near the top), where you can restore it or permanently purge it. So keep your ledger clean without fear — nothing is gone until you purge it from Trash. (The review queue's discarded rows live in their own "discarded" view on the Review Queue page, with the same restore/delete choices; receipts likewise have their own trash view under Tools → Receipts.)

16.Backups & moving computers

The entire application state lives in one folder — the app folder containing:

  • finance.db — every transaction, account, category, budget, reconciliation.
  • imports/ — your archived raw bank CSVs.
  • receipts/ — the receipt inbox and the filed tree.
  • .app_password and .secret_key — your login and session secrets.
  • The code itself.

Backing up

  1. Quit the app (or just don't be mid-import).
  2. Copy the whole folder somewhere safe: an external drive, or a folder that your existing backup system (Time Machine, File History, a synced drive) already covers. That's it — there is no export step, no database dump, no cloud account.
  3. Do it on a schedule you trust; monthly after your close (section 5) is a natural rhythm. Always do it before letting an AI assistant modify the code.
If you sync the live folder to a cloud drive Syncing a live SQLite database while the app is writing can corrupt the synced copy. Prefer copying the folder to the synced location while the app is stopped, rather than running the app from inside a continuously-syncing folder. Also remember: putting your financial data on a cloud drive is a privacy decision — the app itself never does it.

Moving to a new computer

  1. On the old machine: stop the app and copy the whole folder.
  2. On the new machine: install Python 3.10+ and put the folder wherever you like.
  3. In the folder: create the venv and pip install -r requirements.txt (section 2 steps 4–5 — the venv itself doesn't survive the move; recreate it rather than copying it).
  4. Start the app and log in with your usual password.
    Checkpoint Dashboard, ledger, receipts links — all exactly as you left them. Everything is in the folder; nothing lives anywhere else on the old machine.

Restoring from backup is the same operation: replace the folder with the backup copy, recreate the venv if needed, start the app.

17.Troubleshooting

"Address already in use" when starting / can't start the app

Something (almost always an older copy of the app) is still holding port 5005.

  • macOS/Linux: run ./restart.sh — it kills anything on the port, then starts fresh. By hand: lsof -ti tcp:5005 | xargs kill, then start the app.
  • Windows: netstat -ano | findstr :5005 shows the PID in the last column; taskkill /PID <pid> /F kills it; then python app.py.
  • Worst case, restart the computer — inelegant, always works.
The app is running but behaving oddly / my code change isn't showing up

Classic stale-process problem: an old copy of the server is still serving. Open http://127.0.0.1:5005/health in your browser — a small JSON status the app exposes exactly for this. Check started_at (is that when you think you started it?) and pid. If the start time is old, do a hard restart (./restart.sh, or kill by port as above) and re-check /health — the version fingerprint changes when the running code actually changed.

Import says "Parsed 0 importable rows…"

The app never fails an import silently — read the message, it names the cause:

  • "this doesn't look like a checking export (missing the "Posting Date" column)" — you imported a file against the wrong account type (e.g. a credit-card CSV into a checking account, or vice versa). Re-import with the right account selected. The credit-card equivalent checks for "Transaction Date"; the Venmo importer looks for the Venmo statement header (ID / Datetime / Amount).
  • "N row(s) with an unparseable Amount (e.g. "$1,063.03")" — the file was opened and re-saved with currency formatting (usually via Excel). The parser tolerates most formats ($ signs, commas, parentheses-negatives), so this usually means something stranger — open the file in a text editor and look at the quoted example.
  • "N row(s) missing a date/description" — genuinely blank rows; if it's every row, it's the wrong-file case above.
I imported the same statement twice

By design, nothing bad happened: re-imported rows are flagged as Duplicate in the review queue instead of entering your ledger. Confirm the flags (discard them) and move on — or, if the whole batch was a mistake, undo it from Recent Imports on the import page.

Receipt scan extracts nothing / wrong amounts / OCR fails
  • Nothing extracts from PDFs: install poppler (brew install poppler on macOS; poppler-utils elsewhere).
  • Nothing extracts from photos: the OCR backends are tried in order (macOS-native first, then tesseract). Install tesseract (brew install tesseract / your distro's package) if the scan reports OCR failures.
  • Wrong totals from photos: almost always photo quality — glare, crumple, or the total cropped out. Re-photograph flat with the whole receipt in frame. The confidence tiers route doubtful extractions to review rather than misfiling, so a wrong extraction costs you a review click, not a corrupted ledger.
  • Receipt won't match: is the transaction imported and approved yet? Matching runs against active ledger rows. Foreign-currency receipts generally need manual placement.
I forgot my password

Open the app folder and look for the file .app_password (it's hidden — in Terminal: cat ~/personal-financial-tracker/.app_password; in PowerShell: type .app_password from inside the folder). That file is the password. Edit it to set a new one, then restart the app. No file = the password was never changed = changeme.

Anything else

Copy the exact error — from the browser and from the terminal running the app (or app.log if you used start.sh) — and paste it into an AI assistant with a sentence about what you were doing. These are ordinary Python/Flask problems and AI assistants are extremely good at them. If it looks like a bug in the app itself, the maker would like to hear about it: section 18.
